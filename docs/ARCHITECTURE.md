# 🏗️ Архитектура Murnet v5.1

Этот документ описывает внутреннее устройство узла Murnet, его ключевые компоненты и принципы их взаимодействия.

## 🔧 Основные модули

Узел Murnet (`core.node.SecureMurnetNode`) состоит из нескольких независимых, но взаимосвязанных модулей:

1.  **Identity (`core.crypto`)** : Ядро безопасности. Отвечает за создание и хранение криптографической идентичности узла (ключи Ed25519), подписание данных и верификацию подписей.
2.  **Transport (`core.transport`)** : Сетевой уровень. Управляет UDP-сокетами, обеспечивает надежную (с подтверждением) и ненадежную отправку пакетов, защиту от реплеев и базовое шифрование канала (сессионные ключи).
3.  **Routing (`core.routing`)** : Уровень маршрутизации. Реализует Link-State протокол для построения топологии сети и вычисления оптимальных путей до любого узла.
4.  **DHT (`core.murnaked`)** : Уровень распределенного хранилища. Реализует протокол Kademlia (Consistent Hashing) для хранения и поиска данных (сообщения, профили, публичные ключи).
5.  **Storage (`core.storage`)** : Уровень персистентности. Отвечает за сохранение всех данных на диск (SQLite), кэширование (LRU) и управление сроком жизни записей.
6.  **API Server (`api.server`)** : Интерфейс для внешних приложений. Предоставляет REST API и WebSocket для управления узлом и обмена сообщениями.
7.  **Mobile Optimizers (`mobile.*`)** : Набор модулей для адаптации работы узла на мобильных устройствах (экономия батареи, адаптация к типу сети).

## 🔐 Безопасность (Security by Design)

Безопасность в Murnet встроена на всех уровнях, а не добавлена в конце.

### Защита на уровне Криптографии (`crypto.py`)

1.  **Защита от атаки Chalkias**: При подписи данных (`identity.sign()`) метод использует только внутренний приватный ключ, а не принимает публичный ключ извне. Это предотвращает атаку, когда злоумышленник может заставить подписать данные чужим ключом.
2.  **Современные алгоритмы**: Используются только проверенные и современные алгоритмы: Ed25519 (подписи), X25519 (обмен ключами), AES-256-GCM (шифрование), Blake2b (хэширование) и Argon2id (деривация ключей).
3.  **Безопасная обработка ключей**: Приватные ключи хранятся в памяти и при деструкторе объекта предпринимается попытка их затереть.
4.  **Валидация входных данных**: Все входные данные (Base64, JSON) проверяются на длину и корректность перед обработкой, чтобы предотвратить DoS-атаки большими сообщениями.
5.  **Constant-Time Comparison**: Все критические сравнения (HMAC, подписи) выполняются за постоянное время, чтобы предотвратить timing-атаки.

### Защита на уровне Транспорта (`transport.py`)

1.  **Аутентификация пакетов (HMAC)**: Каждый пакет данных сопровождается HMAC-тегом, вычисленным с использованием сессионного ключа. Подделка или изменение пакета невозможна без знания ключа. Пакеты HELLO (начало handshake) не аутентифицированы, но их формат строго ограничен.
2.  **Защита от Replay-атак**: Каждый пакет содержит уникальный номер последовательности (`sequence`) и временную метку (`timestamp`). Транспорт отслеживает полученные номера в скользящем окне, отклоняя повторные пакеты.
3.  **Rate Limiting**: Ограничение на количество пакетов в секунду с одного IP-адреса и глобальный лимит. Это защищает от флуда и простых DDoS-атак.
4.  **Ограничение размера пакета**: Максимальный размер пакета ограничен, чтобы предотвратить атаки, связанные с отправкой гигантских пакетов.

### Защита на уровне DHT (`dht_rpc.py`)

1.  **Аутентификация RPC**: Все удаленные вызовы DHT (store, retrieve) аутентифицируются с помощью HMAC. Отправитель подписывает запрос, используя общий ключ, полученный из его публичного ключа.
2.  **Nonce и Timestamp**: Аналогично транспорту, каждый DHT-запрос содержит уникальный `nonce` и временную метку для предотвращения реплеев.
3.  **Система банов**: При многократных неудачных попытках аутентификации с одного адреса, узел добавляется в бан-лист.

### Защита на уровне Маршрутизации (`routing.py`)

1.  **Подписанные LSA**: Каждый Link State Advertisement (LSA) подписывается узлом-источником. Получатели могут проверить подлинность информации о связях.
2.  **Trust Score**: Каждому соседу присваивается оценка доверия. При получении невалидных LSA (неправильная подпись, странные данные) эта оценка снижается. Маршруты через узлы с низким доверием не используются или используются с более высокой "стоимостью".
3.  **Обнаружение LSA Flood**: Система отслеживает частоту получения LSA от конкретного узла. При превышении лимита LSA игнорируются.

## 📦 Поток данных: Отправка сообщения

1.  **Внешний запрос (API/CLI):** Пользователь вызывает `node.send_message(to, text)`.
2.  **Создание сообщения (`node.py`):** Узел создает словарь с данными сообщения, подписывает его своим приватным ключом. Если у отправителя есть публичный ключ получателя, сообщение шифруется (E2E).
3.  **Сохранение (`storage.py`):** Сообщение сохраняется в локальной БД.
4.  **Постановка в очередь (`node.py`):** Сообщение кладется в `outbox_queue`.
5.  **Маршрутизация (`node._process_outgoing`):** Воркер забирает сообщение из очереди и спрашивает у `routing.get_next_hop(to_addr)` следующий узел.
6.  **Транспортная отправка (`transport.send_to`):** Узел передает сообщение и адрес следующего перехода транспорту.
7.  **Формирование пакета (`transport.py`):** Транспорт создает заголовок `PacketHeader` с номером последовательности и временной меткой, вычисляет HMAC с использованием сессионного ключа для этого пира и отправляет UDP-дейтаграмму.
8.  **Доставка:** Пиры пересылают сообщение по цепочке, пока оно не достигнет адресата.
9.  **Подтверждение (ACK):** Получатель отправляет ACK, который путешествует обратно по тому же пути.

## 📡 DHT: Распределенное хранилище

Murnet использует модифицированную версию Kademlia для хранения данных, которые должны быть доступны всей сети (например, публичные ключи, связанные с адресами, или файлы).

*   **Consistent Hashing:** Кольцо с виртуальными узлами (vnodes) для равномерного распределения нагрузки.
*   **Репликация:** Каждый ключ хранится на `replication_factor` узлах для отказоустойчивости.
*   **Hinted Handoff:** Если целевой узел репликации недоступен, данные временно сохраняются у текущего узла (hint) и будут доставлены позже.
*   **Anti-Entropy:** Периодическая проверка и исправление расхождений в данных между репликами.

## 🔄 Маршрутизация: Link-State

Внутри сети Murnet узлы обмениваются информацией о связях (Link State Advertisements - LSA), создавая полную картину топологии.

*   **Link State Database (LSDB):** Каждый узел хранит карту сети, собранную из LSA.
*   **Dijkstra's Algorithm (SPF):** На основе LSDB узел вычисляет кратчайшие пути до всех известных узлов.
*   **Equal-Cost Multi-Path (ECMP):** При наличии нескольких равнозначных путей, трафик распределяется между ними.
