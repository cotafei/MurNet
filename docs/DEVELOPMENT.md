# 🛠️ Разработка и Тестирование

Это руководство для разработчиков, желающих внести свой вклад в проект Murnet.

## 📋 Настройка окружения

```bash
# 1. Клонируйте форк репозитория
git clone https://github.com/cotafei/MurNet.git
cd murnet

# 2. Создайте виртуальное окружение
python -m venv venv
source venv/bin/activate  # Linux/Mac
# venv\Scripts\activate  # Windows

# 3. Установите зависимости, включая тестовые
pip install -r requirements.txt
pip install pytest pytest-cov pytest-asyncio

```

## 🧪 Тестирование

Проект имеет обширный набор тестов, разделенных по категориям.

### Запуск всех тестов

```bash
pytest
```

### Запуск по категориям

*   **Unit-тесты:**
    ```bash
    pytest tests/unit/ -v
    ```
*   **Интеграционные тесты (могут быть медленными):**
    ```bash
    pytest tests/integration/ -v -m integration
    ```
*   **Тесты безопасности:**
    ```bash
    pytest tests/security/ -v -m security
    ```
*   **Тесты производительности:**
    ```bash
    pytest tests/performance/ -v -m performance
    ```

### Проверка покрытия кода

```bash
pytest --cov=core --cov=api --cov=mobile --cov-report=term-missing
```

## 🧠 Как тестировать безопасность

Многие тесты безопасности находятся в `tests/security/test_security.py`. Они нацелены на проверку конкретных векторов атак.

*   **`test_private_key_not_exposed_in_signature`:** Проверяет, что приватный ключ не "протекает" в подпись.
*   **`test_signature_verification_strict`:** Проверяет, что подпись не проходит при измененных данных.
*   **`test_e2e_encryption_freshness`:** Убеждается, что для одного и того же сообщения каждый раз генерируется разный шифротекст (из-за уникального nonce).
*   **`test_rate_limiting_blocks_flood`:** Проверяет работу rate-limiter'а.
*   **`test_replay_protection_sequence_numbers`:** Проверяет, что повторно отправленный пакет с тем же номером последовательности отклоняется.
*   **`test_lsa_flood_detection`:** Проверяет защиту от флуда LSA-пакетами.

## 📝 Рекомендации по написанию кода

1.  **Безопасность прежде всего:** При добавлении новой функциональности всегда думайте о безопасности. Используйте предоставленные криптографические примитивы, не изобретайте свои.
2.  **Валидация входных данных:** Любые данные, приходящие из сети или от пользователя (через API), должны быть тщательно проверены (длина, тип, формат).
3.  **Используйте контекстные менеджеры:** Для работы с блокировками (`threading.RLock`) и транзакциями БД используйте контекстные менеджеры (`with ...:`), чтобы гарантировать освобождение ресурсов.
4.  **Документируйте сложные участки:** Код, связанный с криптографией или сложной логикой безопасности, должен быть хорошо прокомментирован, чтобы другие разработчики понимали, что происходит и почему.
5.  **Пишите тесты:** Новая функциональность должна сопровождаться тестами. Для критически важных участков — обязательно security-тестами.

## 📤 Как предложить изменения (Pull Request)

1.  Создайте новую ветку от `main`: `git checkout -b feature/your-feature-name`.
2.  Внесите изменения и закоммитьте их с понятными сообщениями.
3.  Убедитесь, что все тесты проходят: `pytest`.
4.  Запушьте ветку в свой форк: `git push origin feature/your-feature-name`.
5.  Откройте Pull Request на GitHub. В описании подробно объясните, что вы сделали и почему.
